## 4.1 Instructions wording

- `Go to X.` / `Click X.` / `Method N:` for options / `Complete the following steps:`
- Split compound instructions: never "Do A and do B" — use two separate steps.
- End complete sentences with `.`

## 4.2 Terminal commands

Precede with a wiki link; use ` ```terminal ``` ` block:

~~~markdown
1. [Run using the `VS Code Terminal`](../../../wiki/vs-code.md#run-a-command-using-the-vs-code-terminal):
   ```terminal
   <command>
   ```

~~~

## 4.3 Command Palette commands

```markdown
1. [Run using the `Command Palette`](../../../wiki/vs-code.md#run-a-command-using-the-command-palette):
   `<command>`
```

## 4.4 Options vs steps

- **Options:** `Method N:` prefix.
- **Steps:** "Complete the following steps:" then ordered list.

## 4.5 Ordered lists

Use `1. 2. 3.` — never `1. 1. 1.`

## 4.6 Little ToC

For long option/step lists, add a `<!-- no toc -->` mini-ToC with `Method N:` links before the sections.

## 4.7 Table of contents

- Insert ToC right after the document title.
- Generated by `Markdown All in One`.
- Use `<h2>Heading</h2>` HTML tags to exclude a heading from the ToC.

## 4.8 Links and cross-references

- Link a concept/tool on its **first mention in each section**; not on subsequent mentions.
- Wiki files cross-reference other wiki files: `[concept](./other-wiki.md#section)`.
- Use relative paths for all links.
- Provide a link to each file that exists in the repo.
- Link format for wiki references from tasks: `[concept name](../../../wiki/<file>.md#<section>)`.
- **Compound phrases:** link the whole phrase as one unit — good: `` [`GitHub` pull request](./github.md#pull-request) ``.

## 4.9 Notes, tips, warnings

Use GFM alerts — **never indent them inside list items**:

```markdown
> [!NOTE]
> ...
> [!TIP]
> ...
> [!IMPORTANT]
> ...
```

## 4.10 Images

```markdown
<img alt="Description" src="../../images/wiki/vs-code/example.png" style="width:400px"></img>
```

## 4.11 Collapsible hints/solutions

```markdown
1. <details><summary>Click to open a hint</summary>

   The hint text here.

   </details>
```

## 4.13 Diagrams

- Format: `.drawio.svg` (editable via draw.io).
- Store in `lab/images/`.
- Reference with `![Alt text](../images/diagram.drawio.svg)`.

## 4.14 `<!-- TODO -->` comments

Use `<!-- TODO description -->` for work-in-progress. Tracked by `Todo Tree` extension.

## 4.15 `<!-- no toc -->` comments

Put `<!-- no toc -->` before a list to exclude it from the auto-generated ToC.

## 4.16 Code snippets in explanations

Use fenced code blocks with the language specified when walking through code.

## 4.17 Heading levels in section titles

When specifying sections to create, include the heading level: `## Section Name`.

## 4.18 Inline formatting of technical terms

Wrap tool names, technical terms, and acronyms in backticks: `` `VS Code` ``, `` `Git` ``, `` `Docker` ``, `` `Python` ``, `` `SQL` ``.

## 4.20 Placeholders in docs

Use placeholders instead of hardcoded values (e.g., `<pgadmin-url>` not `http://127.0.0.1:5050`).
Define every placeholder in `wiki/placeholders.md`.
